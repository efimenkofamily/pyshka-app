# Архитектура проекта «Клуб Пышка»
## Техническая документация, структура БД и подсистем (Релизная версия 4.5 — Fullstack App)

Этот документ содержит актуальное описание архитектуры приложения, структуры таблиц в базе данных Supabase, логики парсера, конфигураций маржинальности, десктопной веб-админки, клиентского SPA-приложения на React и подсистемы маршрутизации алертов.

---

## 📂 1. Структура репозитория и файлы проекта

* `/client` — Клиентское WebApp-приложение для Telegram-бота (SPA на React + Vite). Включает в себя каталог, интерактивную корзину, профиль пользователя с плавными анимациями, систему оформления заказов, мобильную админку и стартовую анкету. Деплоится на Netlify (директория `dist`).
* `/admin` — Десктопное SPA-приложение на React + Vite для разработчиков и администраторов. Содержит продвинутый журнал заказов с функционалом ручной модификации чеков, калькуляторы прибыли, страницу изолированного управления уведомлениями и маржи. Деплоится на Netlify (базовая директория `admin`).
* `Google Apps Script` — Внешний бэкенд-парсер прайс-листов из Google Таблиц в базу данных Supabase.
* `Supabase Edge Functions` — Бессерверный реактивный слой бэкенда. Содержит функцию `notify-status` (маршрутизация транзакционных пушей) и `sync-catalog` (автоматическая синхронизация каталога по API).

## ⚙️ 2. Логика работы систем парсинга и синхронизации

### 2.1 Парсер прайс-листов (Google Apps Script)
Парсер работает по следующим принципам:
* **Цветовая маршрутизация (HEX):** Скрипт читает цвета ячеек в Google Таблицах. По цвету определяется иерархия: маркер Производителя (`#434343`), маркер Линейки (`#fff2cc`). Черные строки (например, ice-версии) аппаратно игнорируются (`#000000`).
* **Умный парсинг расходников (RegEx Расщепление):** В таблицах поставщика расходники часто записаны в одну строку. Парсер автоматически "расщепляет" такие строки на самостоятельные позиции в БД:
    * Название модели устройства записывается в колонку `series` (Линейка).
    * Сопротивления и объем бака извлекаются через регулярные выражения и записываются в колонку `flavor` (Вкус).

### 2.2 Парсер API Поставщика (Edge Function: `sync-catalog`)
Автономный скрипт на Deno для прямой синхронизации остатков и оптовых цен с сайтом поставщика.
* **Обход Cloudflare (Стелс-режим):** Функция эмулирует поведение настоящего браузера, передавая заголовки `User-Agent` и `Referer`, а также перехватывает и подставляет `Cookies` из ответа сервера для успешной авторизации.
* **Умная очередь и Анти-429:** Во избежание блокировок IP-адреса, запросы к карточкам товаров идут строго последовательно с микро-паузой. В случае получения статуса `429 Too Many Requests`, парсер автоматически замирает на 2 секунды и повторяет попытку (до 3 раз).
* **RLS Bypass:** Для бесшовного удаления и записи новых данных в БД используется сервисный административный ключ `SUPABASE_SERVICE_ROLE_KEY`.

## 🎛 3. Матрица и Маршрутизация Уведомлений (Telegram Алерты)

Система алертов полностью отвязана от монолитного кода приложений и работает реактивно на базе триггерных вебхуков Supabase и бессерверной функции Edge Function `notify-status`.

### Логика работы подсистемы:
1. **База данных (Database Webhooks):** При совершении операций `INSERT` или `UPDATE` в таблицах `users`, `orders` или `config`, СУБД генерирует payload и отправляет его по протоколу POST в Edge Function.
2. **Сканирование конфигурации:** Функция первым делом запрашивает поле `notification_matrix` из таблицы `config` и кэширует правила доступа в памяти.
3. **Фильтрация по ролям (Вахтер):** Для каждого получателя вычисляется его текущая роль (`developer`, `admin`, `approved`, `pending`). Если матрица содержит `false` для связки `[Системное событие][Роль]` — отправка сообщения блокируется.

### Список системных событий матрицы:
* `NEW_USER_PENDING` — Оповещение администрации о поступлении новой заявки на вступление в клуб.
* `NEW_ORDER` — Отправка структурированного чека и подтверждения клиенту в ЛС.
* `NEW_ORDER_STAFF` — Мгновенное дублирование состава и метаданных нового заказа в ЛС администрации.
* `ORDER_STATUS_CHANGED` — Информирование покупателя об изменении статуса его заказа менеджером.
* `STORE_OPENED` — Автоматический массовый броадкаст-пуш об открытии магазина при изменении `open_date`.
* `STORE_CLOSED` — Массовый броадкаст-пуш о закрытии магазина при обновлении `close_date`.

## 🗄 4. Структура Базы Данных и Автоматизация (Supabase)

### Автоматизация (Cron Jobs)
В базе данных активированы расширения `pg_net` и `pg_cron`. Настроен внутренний таймер `sync-catalog-hourly`, который каждый час в 00 минут автоматически отправляет POST-запрос на вызов функции `sync-catalog`, обеспечивая актуальность склада без участия администратора.

### Таблица: `users`
| Колонка | Тип данных | Описание |
| :--- | :--- | :--- |
| **id** | BigInt | Telegram ID пользователя (Primary Key) |
| **username** | String | Юзернейм в Telegram (@nickname) |
| **status** | String | Роли: `pending`, `approved`, `admin`, `developer`, `blocked` |

### Таблица: `orders`
| Колонка | Тип данных | Описание |
| :--- | :--- | :--- |
| **id** | UUID | Идентификатор заказа (Primary Key) |
| **user_id** | BigInt | Telegram ID покупателя |
| **items** | JSON | Состав заказа (Map: UUID товара -> объект с `qty` и параметрами `product`) |
| **total_price** | Number | Итоговая розничная стоимость с доставкой |

### Таблица: `products`
| Колонка | Тип данных | Описание |
| :--- | :--- | :--- |
| **id** | UUID | Уникальный идентификатор товара (Primary Key) |
| **category** | String | Категория товара |
| **manufacturer**| String | Производитель / Бренд |
| **series** | String | Линейка товаров |
| **flavor** | String | Вкус / Описание позиции |

| **stock_qty** | Number | Фактический остаток на складе поставщика |

### Таблица: `series_images`
Отвечает за хранение фотографий линеек в отрыве от динамических UUID товаров из таблицы `products`.
| Колонка | Тип данных | Описание |

| **notification_matrix** | JSONB | Матрица доступов и маршрутизации Telegram пушей по ролям |

## 💻 5. Функциональные модули Десктопной Админки (`/admin`)

* **Авторизация (`Login.jsx`):** Интеграция с Telegram Login Widget. Доступ открывается исключительно для статусов `developer` и `admin`.
* **Диспетчер заказов (`Orders.jsx`):** Интерактивный журнал продаж со сквозной аналитикой и расширенным контекстным управлением.
    * **Двухколоночный пульт модификации:** Инструмент для ручного урегулирования пересортицы. Содержит реактивную систему поиска замен по актуальной номенклатуре склада.
    * **Финансовый процессор:** При редактировании чека «на лету» санитизирует закупочные цены, применяет формулы маржинальности из `config` и рассчитывает новую сумму.
* **Каталог товаров и маржа (`Products.jsx`):**
    * **Динамическое ценообразование:** Рассчитывает розницу на базе `config` и показывает плановую прибыль с единицы товара.
    * **Умный менеджер фото (Smart Binding):** Прикрепление изображений осуществляется через модальное окно с поиском по таблице `series_images`. Это позволяет переиспользовать уже загруженные фото после обновлений БД, так как привязка идет по текстовым полям `manufacturer` и `series`, а не по изменчивым UUID.
    * **Категоризация:** Имплементированы визуальные бейджи категорий и динамический `<select>` для фильтрации таблицы по товарным группам.
* **Управление Уведомлениями (`Notifications.jsx`):** Изолированная страница-пульт управления алертами. Сохраняет настройки матрицы в `config` атомарно, исключая перезапись финансовых полей.

## 📱 6. Клиентское WebApp приложение (`/client`)

### Основные компоненты и модули:
* **Основной контроллер (`App.jsx`):** Управляет авторизацией, навигацией и состояниями. Блокирует каталог для обычных покупателей вне расписания дропов.
* **Умный Каталог:**
    * **Светофорные бейджи крепости:** Зеленый фон для 0-20 мг, оранжевый для 21-40 мг, красный для 41+ мг.
    * **Плавные Анимации:** Раскрытие списков вкусов переведено на производительную CSS Grid анимацию.
* **Оформление заказа (`Checkout.jsx`):**
    * **Интерактивная корзина:** Инкремент и декремент количества товаров (`+` / `-`) непосредственно в корзине.
    * **Swipe-to-Delete:** Нативный паттерн удаления позиций — по свайпу справа налево выезжает кнопка удаления.
* **Профиль пользователя (`Profile.jsx`):**
    * **Дизайн чека:** Двухстрочный паттерн отображения товаров с разделительными линиями.
    * **Кнопка изменения:** Позволяет отменить заказ в БД, вернуть товары в корзину и перейти в каталог для редактирования.
* **Система резервирования остатков (Inventory Hold) и Realtime:**
    * **Умный расчет:** Доступное количество товара вычисляется по формуле: `Остаток поставщика` минус `Товары в активных заказах других пользователей` минус `Товары в текущей корзине`.
    * **Визуальная обратная связь:** Под кнопками количества динамически выводится подсказка ("Осталось: N шт."). Если лимит исчерпан, кнопка "+" блокируется.
    * **WebSockets Синхронизация:** Приложение слушает канал `postgres_changes` таблицы `orders`. Как только кто-то оформляет заказ, у всех остальных клиентов мгновенно пересчитываются и обновляются остатки без перезагрузки страницы.
